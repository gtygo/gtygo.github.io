如何写好代码？

发表于 2019-09-03 | 分类于 个人感悟

序

写出优雅的代码相信是所有工程师共同的目标，但是这个目标需要工作经验，以及踩过的坑进行总结和反思，然后我的编程并不多，本片文章即是对自己毕业数月的总结，也是对未来编程的一个方法论

如何写好代码？

抽象

一个好的设计可以适配多种不同的情形，同样好的设计拥有着更优秀的扩展性和可用性，我认为在设计的过程中要多思考需求之外的东西—–抽象，需要脱离实际的需求去思考我的实现能否在其他场景中使用？ 如果未来某天代码需要重构/别人接手需要增加更多的需求时，是否能够让人非常轻松的完成任务？

拆解

拆解就是把一个复杂的需求，在逻辑层面进行拆分，更有利于我们思考，比如我要设计一个双向美元兑换人民币的服务，那我认为首先可以拆成以下几个小的模块：

  1. 汇率 ：我们需要一个稳定高可用并且可以精确到秒级的获取汇率API
  2. 资金池 ： 我们需要两个资金足够充足的银行账户，用来提供资金兑换
  3. 代码复用 ： 双向兑换，则要考虑是否能够做成无论哪个方向都走一套逻辑？ 以及如果有一天需要添加更多的兑换币种，是否能够做到开发成本最低化？
  4. 事物 : 涉及到钱的安全性，是否能够在发生意外情况（资金池没钱、断电、地震、海啸）等因素保持ACID？
  5. 安全 : 如何保证我们的兑换是安全的？ 用户人民币账户和美元账户是不是要做成一对一绑定？ 如果绑定后什么时候解绑定？
    当然，一个实际真正的兑换业务肯定要比上述来的复杂得多，这里只是说了一下方法论，主要的思想就是大事化小，用更小的粒度去思考。

多使用/学习优秀的库

没错，言外之意就是少造轮子，虽然造轮子可以提高自身对于某个技术的理解，但是我认为在工作中为了完成一个需求去造某类轮子是得不偿失的，第一 ：写的并不一定别开源的好，第二 ：维护成本高，文档也不够全面，可能存在的坑自己都不知道。 当然，如果是为了学习某个技术，自己动手去实现可以最大程度的掌握这个技术存在细节。
