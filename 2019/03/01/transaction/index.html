UTXO模型下的交易处理

发表于 2019-03-01 | 分类于 区块链技术

UTXO模型的交易具有很多特点，本文主要针对一些关键的技术点进行分析。

数据结构

交易的核心数据结构如下：
-transaction hash
-vin
–outpoint
—-previous hash
—-index
–script sig
-vout
–value
–script pubkey
每次交易需要找到vin ，也就是对应上一笔交易产生的vout，通过previous hash 和index 可以很容易的定位到前一笔交易的vout，但是，整个区块链系统不是线性执行，交易可以是从本地节点产生，也可以是通过p2p网络获取的，这时会出现一种情况就是当我收到交易时，在我本地找不到上一笔交易的vout，那么我们可以认为这是一个orphan transaction，稍后会对此进行说明。

经典的三部曲

使用UTXO模型使用这下面的三个步骤来完成一个交易的处理，当然，这三个步骤只是对于用户/RPC层级可见的，步骤如下。

  1. create raw transaction 创建一个未签名的hex格式编码的交易序列
  2. sign raw transaction 对一个原始的交易序列进行签名（这一动作必须是在本地完成，不能在server端涉及到密钥的操作）
  3. send raw transaction 将一个签名的交易序列发送到transaction pool 等待矿工打包上链。

上面我们可以看到交易的产生于使用，通过这三种方式将交易进行创建及处理。

孤儿池

当找不到前一笔交易的vout时，将这个交易打入孤儿池，等待p2p网络接收到其上一笔交易。 这里的孤儿池主要是解决因网络延迟等原因不能够及时接受到交易的问题。
